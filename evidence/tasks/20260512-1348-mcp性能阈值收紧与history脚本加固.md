# MCP 性能阈值收紧与 history 脚本加固

## 任务目标

- 用真实 CI performance artifact 历史收紧 `scripts/check_mcp_performance_gate.py` 的 `THRESHOLDS`。
- 保持 Streamable HTTP adapter 只作为协议 adapter，集成测试继续复用真实 adapter contract，不复制业务逻辑。
- 保持 `reports_build` / `figures_research` 的 Tasks/progress 未来契约同时覆盖 cancel、progress、result resource。
- 继续保持无前缀工具名，不恢复 `keepa.*` 旧名。

## 背景与输入

- 本轮不访问真实 Keepa API，不消耗真实 token；所有 MCP 验证均为 fixture / 本地协议层。
- 已有两轮 `mcp-sdk-adapter` job 成功上传 performance artifact，但其中整体 workflow 曾因 Windows 单测失败或 snapshot 漂移失败。
- 新提交 `f77233c` 修复 Windows history CLI 后，CI run `25715918950` 全部通过，并上传第三份可用 artifact。

## 处理过程

- 修复 `scripts/summarize_mcp_performance_history.py`：
  - CLI 启动时强制 stdout/stderr 使用 UTF-8，避免 Windows runner 在 `ensure_ascii=False` JSON 输出中文 `next_actions` 时触发编码失败。
  - `_load_reports()` 跳过不可读、非 UTF-8、非法 JSON 或不含 `benchmarks` 的 JSON 文件，避免 artifact 目录混入 summary / notes / partial JSON 时破坏汇总。
- 加固 `tests/test_mcp_performance_history.py`：
  - subprocess 使用脚本绝对路径、固定 `cwd=REPO_ROOT`、`encoding="utf-8"`。
  - 显式模拟 `PYTHONIOENCODING=cp1252`，覆盖 Windows 非 UTF-8 环境。
  - 增加非 performance JSON 与非法 JSON fixture，确认 history CLI 会跳过噪声输入。
  - tighten 逻辑测试改为临时注入宽松阈值，避免生产阈值收紧后测试假设失效。
- 通过 GitHub Actions 下载三份真实 artifact：
  - `25715918950` / `f77233c`：整体 CI 成功，`mcp-sdk-adapter` 成功上传。
  - `25715728619` / `a62b193`：`mcp-sdk-adapter` 成功上传，整体 workflow 因旧 Windows history CLI 失败。
  - `25715600132` / `f3b10fb`：`mcp-sdk-adapter` 成功上传，整体 workflow 因旧 schema snapshot 失败。
- 使用 `scripts/summarize_mcp_performance_history.py <artifact-dir> --json --out <summary> --min-samples 3` 汇总，结果 `report_count=3`、`ready_to_tighten=true`、`tighten=19`、`watch_regression=0`。
- 按 history 脚本建议收紧 `THRESHOLDS`：
  - latency 类阈值统一收紧到脚本保底值 `10.0 ms`。
  - bytes 类阈值按真实 CI 历史 max * `1.2` 且不低于 `1024` 回写。

## 风险与后续

- 本次阈值来源是 `mcp-sdk-adapter` job 的 Linux Python 3.11 artifact；其中两份 artifact 所属 workflow 整体失败，但 artifact job 本身成功。后续可继续累计全绿 workflow artifact，再二次复核是否需要微调。
- `p95_ms=10.0` 是脚本保底阈值，当前 CI / 本地均远低于该值；如果未来 CI runner 抖动触发失败，应先检查原始 artifact 与代码变化，不直接放宽门禁。
- Streamable HTTP 仍不能复制业务逻辑；真实 HTTP adapter 落地时必须继续复用 `handle_mcp_message` / `AgentSession` / service 层。
- Tasks/progress 目前仍是 future contract；真正实现时需要同步设计 `tasks/cancel`、`notifications/progress`、`tasks/result` 与 `keepa://tasks/{task_id}/result` 结果资源。
- Keepa token 不足仍按等待/降级处理，不作为永久拒绝；普通 `tools/call` 不应因等待 token 或长任务而无限阻塞。

## 结论

Performance gate 已从宽松基线收紧为真实 CI artifact 历史驱动阈值；history 汇总脚本修复了 Windows 编码与目录噪声输入问题；HTTP adapter 与 Tasks/progress 契约继续保持协议边界和长任务边界，不影响现有 stdio 生产服务。
