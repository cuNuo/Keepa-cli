# MCP builder 官方实现对照评估

## 任务目标

评估 Keepa-cli 当前 MCP 实现相较于 `$mcp-builder` 官方实现指南是否还需要继续完善，并在允许破坏性重构的前提下给出后续架构建议。

## 背景与输入

- 用户输入：要求思考当前项目相较 `$mcp-builder` 官方实现是否需要继续完善，且可接受破坏性重构。
- 项目当前事实：已存在生产 raw stdio MCP handler、官方 Python SDK low-level adapter、Streamable HTTP adapter、JSON Schema 入参校验、toolset/profile/cursor、resources/templates/prompts、质量门禁与 fixture。
- 对照材料：`D:\.codex\skills\mcp-builder\SKILL.md`、`reference\mcp_best_practices.md`、`reference\python_mcp_server.md`、项目 Serena MCP 相关记忆、`docs/architecture/mcp-python-sdk-adapter-comparison.md`。

## 处理过程

1. 读取 Serena 记忆，确认 MCP 官方 SDK adapter、实现规范审查、Streamable HTTP adapter、Schema validator 与无前缀工具契约的最新状态。
2. 读取 `$mcp-builder` 指南与最佳实践，提取官方实现关注点：官方 SDK、schema/structuredContent、annotations、分页、上下文控制、可操作错误、Streamable HTTP 与评测。
3. 定位当前代码边界：`keepa_cli/agent/mcp.py` 仍是生产 raw JSON-RPC handler；`keepa_cli/agent/mcp_sdk_adapter.py` 是官方 SDK low-level adapter；`keepa_cli/agent/mcp_http.py` 与 `mcp_http_contract.py` 是 HTTP 协议 adapter。
4. 对照现有文档和质量门禁，判断继续完善方向应从“补能力”转为“收敛协议层与降低 drift”。

## 风险与后续

- 风险 1：继续长期保留 raw stdio 与 SDK adapter 双轨，会让协议行为、错误映射、分页和客户端展示产生漂移。
- 风险 2：直接切到 FastMCP decorator 可能损失当前动态 toolset/profile/cursor/resource template 能力。
- 风险 3：当前 HTTP adapter 按稳定协议实现；若 MCP draft 改动正式落地，需要通过 fixture 迁移，不能静默兼容两套语义。

## 结论

需要继续完善，但重点不是继续堆新工具，而是破坏性收敛为“一个 MCP protocol core + 多 transport 外壳”。推荐先把 raw handler 的方法分发、分页、错误映射、结果封装抽到共享 protocol core，再让 stdio、SDK adapter、HTTP adapter 都调用该 core；随后评估是否把官方 SDK low-level adapter 提升为默认 stdio 入口。FastMCP decorator 只适合作为只读样例或未来局部减样板手段，不建议直接替换当前动态 MCP 契约。
