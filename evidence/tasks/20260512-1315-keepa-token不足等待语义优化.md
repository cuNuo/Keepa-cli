# Keepa token 不足等待语义优化

## 任务目标

- 修正 Keepa token 不足语义：token 按套餐速率持续补充，不能把余额不足当作永久拒绝。
- 在 CLI、Agent session、MCP structured error 与 HTTP 429 路径中提供可执行等待、重试与降级建议。
- 保持现有服务入口正常，不访问真实 Keepa API，不恢复旧 `keepa.*` 工具名前缀。

## 背景与输入

- 用户明确指出 Keepa token 每分钟后台生成，生成速率取决于套餐配置。
- 现有 `client.py` 已会对 429 的 `refillIn` 最多自动等待一次，但失败后的错误详情只暴露 `retry_after_ms`，Agent 缺少下一步策略。
- 高成本确认 `confirmation_required` 与真实 429 / `not_enough_token` 必须保持区分：前者是显式确认门禁，后者是余额/速率不足。

## 处理过程

- 在 `keepa_cli/token_budget.py` 新增 `build_token_refill_guidance()`，统一生成：
  - `status_command=tokens.status`
  - `wait_strategy`
  - `estimated_tokens` / `worst_case_tokens`
  - `tokens_left`、`token_deficit`、`refill_in_ms`、`refill_rate_per_minute`
  - `retry_after_seconds`
  - `next_actions` 与命令族降级 `hints`
- 在 `keepa_cli/client.py` 的 HTTP 429 错误路径接入该 guidance：
  - 优先使用 Keepa 响应体 `refillIn`
  - 缺失时回退 `Retry-After` header
  - 仍保留最多自动等待一次、最多 60 秒的现有策略
- 在 `keepa_cli/commands/common.py` 与 `keepa_cli/agent/session.py` 的 `confirmation_required` details 中加入同一 guidance，让 CLI、stdio、MCP tool error 都能提示等待/降级路径。
- 更新 README、README.zh-CN、`docs/agent-contract.md`、`docs/keepa-official-api-notes.md`，明确 token 不足不是永久拒绝。

## 风险与后续

- 当前不会无限等待 token refill；真实 429 仍只自动等待一次，避免普通 tools/call 长时间阻塞。
- `refillRate` 被记录为 `refill_rate_per_minute`，后续若官方字段语义变更，需要同步调整等待估算。
- 长任务仍应走后续 MCP Tasks/progress 设计，并同时支持 cancel、progress、result resource。

## 结论

本轮完成 token 不足等待语义统一：真实 429 会暴露等待秒数与降级建议，高成本 confirmation 会提示 `tokens.status`、等待补充、缩小请求范围或使用离线路径；现有 CLI/MCP/Node 入口验证通过。
