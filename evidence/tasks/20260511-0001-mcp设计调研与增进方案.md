# MCP 设计调研与 Keepa-cli 增进方案

## 任务目标

- 使用用户指定的 `find-skills` skill，调研优秀 MCP 设计模式。
- 对照 GitHub MCP、Serena MCP、DeepWiki、Context7、Filesystem MCP 与 MCP 官方协议能力，提炼 Keepa-cli 后续增强方向。
- 输出面向后续“调研 Agent”接入的增进方案。

## 背景与输入

- 用户要求：技能必须放在本项目下，不放全局路径。
- 本轮只读取全局 `find-skills` skill 作为用户显式指定的调研方法，不写入全局 skill。

## 外部设计模式调研

### MCP 官方协议

- Tools 适合模型主动执行动作；结果可同时返回 `structuredContent` 和 text content。
- Tool annotations 可表达只读、破坏性、幂等、是否访问开放世界等风险信号。
- Resources 用于共享文件、schema、数据库结构、应用上下文等资料；resource templates 用 URI 模板暴露参数化资源。
- Prompts 用于暴露可复用工作流起手式，而不是把长说明塞入 tool 描述。

参考来源：

- https://modelcontextprotocol.io/specification/2025-06-18/server/tools
- https://modelcontextprotocol.io/docs/concepts/resources
- https://modelcontextprotocol.io/specification/2025-06-18/server/resources
- https://modelcontextprotocol.io/specification/2025-06-18/server/prompts

### GitHub MCP

可借鉴点：

- 大量工具按 toolset 分组，默认只启用常用集合，减少上下文污染。
- 支持 `--toolsets`、`--tools`、`--exclude-tools`、`--read-only` 与 dynamic tool discovery。
- 只读模式优先级高：即使显式请求写工具，只读模式下也跳过写工具。
- toolsets 不只影响 tools，也可以连带相关 resources 与 prompts。

对 Keepa-cli 的启发：

- 当前已有 toolsets，但还缺“调研 Agent profile”级别的组合策略，例如 `research-safe`、`research-live-approved`、`fixture-curation`。
- 可以新增 exclude-tools / allowed-tools 层，便于在不同 Agent 中复用同一 MCP server。
- 可以把“只读/离线/可真实消耗 token”做成启动策略，而不仅靠单次 tool 参数。

参考来源：

- https://github.com/github/github-mcp-server
- https://docs.github.com/copilot/how-tos/provide-context/use-mcp/configure-toolsets

### Filesystem MCP

可借鉴点：

- 以 allowed directories / Roots 作为访问边界。
- 提供 `list_allowed_directories`，让 Agent 在读写前确认边界。
- 支持 client roots 动态更新，且没有可用根目录时拒绝运行。

对 Keepa-cli 的启发：

- 当前 chunk/output resource 已限制项目根或临时目录；可进一步暴露 `keepa.context_roots` / `keepa://context/roots`，明确当前允许读取、允许写入、禁止提交的路径集合。
- 对 `chunks_dir`、`out`、`output` 类参数统一做 root policy 与 provenance 标记。

参考来源：

- https://github.com/modelcontextprotocol/servers/blob/main/src/filesystem/README.md
- https://github.com/modelcontextprotocol/servers/blob/main/src/filesystem/index.ts

### Context7 MCP

可借鉴点：

- 两段式工作流：先 `resolve-library-id`，再 `query-docs`。
- 先把自然语言意图解析成稳定 ID，再按 ID 查询高相关文档，降低误命中。
- 返回候选时带 source reputation、snippet 数、版本等选择信号。

对 Keepa-cli 的启发：

- 对 Amazon / Keepa 调研可引入类似两段式：
  - `keepa.resolve_research_target`：把关键词、ASIN、UPC、类目名、seller id 解析为稳定 target candidates。
  - `keepa.query_research_context`：基于 target id 查询 fixtures、schema、evidence、zread、cached outputs、recent graph。
- 对 category / ASIN / seller 的后续工具调用，优先使用 resolved target id，减少 Agent 自己拼错参数。

参考来源：

- https://context7.com/docs/api-guide
- Context7 MCP docs：`resolve-library-id` 与 `query-docs` 两段式工具。

### Serena MCP

可借鉴点：

- project-first：先激活项目，再使用项目记忆和语义工具。
- memory-first：把稳定事实写入长期 memory，一次性过程写 evidence。
- 符号化工具减少整文件读取，并区分 exposed tools 与 active tools。
- 不可用工具不应静默执行，应返回结构化错误。

对 Keepa-cli 的启发：

- Keepa 的调研 Agent 也应 project/session-first：先读取 `keepa://tools/index`、`keepa://prompts/index`、`keepa://evidence/recent`，再决定 toolset。
- 增加“active policy”层：同一个 tool 暴露给客户端，但在 session policy 下可能返回 `inactive_tool`，并解释需要哪个 profile/确认/环境变量。
- 把每次 research graph merge 后的稳定结论写成可读 memory-like resource 或 export，供后续调研 Agent 复用。

参考来源：

- https://github.com/oraios/serena
- DeepWiki: https://deepwiki.com/oraios/serena

### DeepWiki MCP

可借鉴点：

- 将仓库组织为 wiki 页、索引、架构说明和专题页面，适合 Agent 做粗粒度理解。
- 先读 overview / toc，再按专题深入，避免全仓库扫描。

对 Keepa-cli 的启发：

- 项目已有 zread wiki MCP resources，可继续把 Keepa research graph、fixture manifest、Agent eval 结果生成“调研知识页”。
- 调研 Agent 不应每次重跑产品研究；应优先读近期 graph/evidence/wiki 摘要，再决定是否调用 Keepa API。

参考来源：

- https://deepwiki.com/github/github-mcp-server
- https://deepwiki.com/upstash/context7
- https://deepwiki.com/modelcontextprotocol/servers

## 关键缺口

1. 缺少 session/profile 级策略层。
   当前 toolset 已能减小 schema 面，但还没有把“fixture-only、offline-first、live-approved、report-only、skill-authoring”等调研 Agent 场景固化成 policy。

2. 缺少 Context7 式 target resolution。
   现在 Agent 可以直接调用 category/product/finder 工具，但“用户输入到底是 ASIN、UPC、类目、seller、关键词还是已存在 evidence”仍需要 Agent 自己判断。

3. 缺少 Filesystem 式 roots 可见性。
   资源读取已有路径限制，但 Agent 不能先查询“本 MCP server 当前允许读写哪里、哪些路径不能提交、哪些目录只是 runtime logs”。

4. 缺少 GitHub MCP 式工具排除与动态启用。
   `tools/list` 支持 toolset，但没有 per-session `exclude_tools`、`allow_tools`、dynamic enable/disable toolset。

5. 缺少调研 Agent 的可复用技能落点。
   `.codex/skills/keepa-agent-research` 已有基础工作流，但可以继续演进成“Agent 先读资源、解析目标、规划预算、执行、合并图谱、写报告”的完整 SOP。

## 增进方案

### P0：调研 Agent 接入前的安全与发现面

1. 新增 `keepa://context/policy` resource。
   内容包括：默认 mode、toolsets、read/write roots、runtime log 禁提交目录、是否允许 live Keepa、是否存在 `KEEPA_API_KEY`（只返回布尔，不返回值）、token budget 默认阈值。

2. 新增 `keepa.policy_status` tool 或 docs resource fallback。
   给不支持 `resources/read` 的客户端读取同样信息；保持只读、本地计算。

3. 在 `tools/list` 支持可选 `allow_tools` / `exclude_tools` 参数。
   与 GitHub MCP 对齐，便于调研 Agent 只暴露本轮需要的少数工具。

### P1：Context7 式两段式研究入口

1. 新增 `keepa.resolve_research_target`。
   输入 `query`、`domain`、`hint_type`，输出候选 target：
   `asin`、`category`、`seller`、`keyword`、`fixture`、`evidence`、`graph_node`。

2. 新增 `keepa.query_research_context`。
   输入 resolved target id 与 question，优先查询本地 fixtures、evidence、zread、schema、recent graph、session cache，不默认访问 Keepa。

3. 更新 prompts：
   新增 `keepa.research_agent_start`，强制顺序为 policy -> tools index -> target resolution -> workflow plan -> budget audit -> execution -> graph merge -> report。

### P1：动态工具与 active policy

1. 引入 session policy：
   - `offline_fixture_only`
   - `dry_run_default`
   - `live_read_allowed`
   - `tracking_readonly`
   - `fixture_curation`

2. tool 调用时若违反 policy，返回结构化 `inactive_tool` / `confirmation_required`，包含可执行 next action，而不是只给 prose。

3. 对 `audit`、`reports`、`docs`、`tracking-readonly` 之外的潜在写类工具继续默认不暴露。

### P2：面向调研 Agent 的知识产品

1. 新增 `research_brief.export` / `keepa.research_brief_export`。
   输入 merged graph 或多份 payload，输出稳定 briefing：
   `decision_summary`、`risk_summary`、`entity_graph_summary`、`follow_up_plan`、`evidence_links`。

2. 新增 `keepa://research/{id}/brief` 与 `keepa://research/{id}/graph` resource template。
   让后续 Agent 通过 ID 复读研究结果，而不是读整份 JSON。

3. 将 Agent eval fixtures 扩展为“完整调研链路”：
   target resolve -> workflow plan -> fixture tool calls -> graph merge -> brief export。

### P2：项目内 skill 增强

1. 只在项目内更新 `.codex/skills/keepa-agent-research/SKILL.md`。
2. 新增章节：
   - MCP policy/status 首读规则。
   - target resolution 两段式。
   - research brief export。
   - live -> sanitize/promote -> eval parity。

## 三个替代路径

1. 最小路径：只加 `keepa://context/policy` 与 prompt，不新增工具。
   成本最低，适合先让调研 Agent 稳定遵守 offline-first。

2. 标准路径：增加 policy resource、target resolution、query context、research_agent_start prompt。
   最适合近期接入调研 Agent，能明显减少误调用和上下文浪费。

3. 完整路径：再加入 dynamic tool gating、research brief export、完整链路 eval。
   成本较高，但可把 Keepa-cli 从“工具集合”推进为“调研 Agent 后端”。

## 风险与假设

- 外部资料存在版本漂移；本轮使用 2026-05-10/11 可访问公开资料与当前本地代码状态。
- GitHub MCP 的 dynamic tool discovery 仍属于部分环境/本地模式能力，不能假设所有 MCP 客户端都支持。
- 不同 MCP 客户端对 resources、prompts、structuredContent 支持不一致；因此 Keepa-cli 仍需保留 docs tools 与 text fallback。
- 调研 Agent 未来若会进行真实 Keepa 请求，必须保留 explicit confirmation、dry-run、fixture replay 与 token ledger。

## 结论

Keepa-cli 当前 MCP 基础已经扎实；下一步不应优先增加更多低层工具，而应围绕“调研 Agent 接入”补三层能力：

1. policy/status：让 Agent 先知道边界。
2. target resolution：把自然语言或模糊输入转换为稳定研究目标。
3. research brief：把多工具输出压成可复用的研究知识产品。
