## 风险与后续

- 当前风险：`keepa_MCP` 中部分销售速度、库存和利润公式是启发式估算，直接移植会降低 Keepa-cli 当前可审计性；README 中存在工具数量表述不一致；`keepa_MCP` 默认 live API 依赖更强，不适合照搬到离线优先主线。
- 建议后续动作：
  - P0：在 Keepa-cli 增加 `seller-metrics` / `velocity` / `inventory` 命令族或 Agent view 字段，把 `monthlySold`、seller count、库存风险、现金流指标结构化输出。
  - P1：增加 MCP prompt / workflow 模板，提供 `inventory-audit`、`velocity-research`、`market-opportunity` 等业务场景入口。
  - P1：为 category ID 和 marketplace 手册提供 `keepa://guides/categories` / `keepa://guides/marketplaces` resources，减少用户查文档成本。
  - P2：设计公式来源与置信度字段，所有估算必须带 `method`、`inputs`、`confidence`、`evidence_path`。

## 结论

- `keepa_MCP` 不应作为工程架构模板照搬，但它很好地暴露了 Keepa-cli 下一步要补的产品化方向：把现有强结构、强治理的 Agent 后端，包装成更贴近卖家决策语言的 MCP 工具、prompt 和 report workflow。
