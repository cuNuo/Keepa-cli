## 任务目标

- 回应“像 zread / Claude Code 一样用前端框架”的交互诉求，把简陋标准库 TUI 升级为现代终端框架工作台。
- 保留旧标准库 TUI 作为缺少依赖、管道输入和低能力终端的 fallback。
- 不改变 Agent JSON / stdio 行为，不让默认路径访问真实 Keepa API。

## 背景与输入

- 用户反馈：当前 TUI 太简陋，没有美感，希望参考 zread、Claude Code 的终端界面体验。
- 技术取舍：选用 Python 生态 Textual/Rich 风格原生终端框架，而不是引入 Vite/浏览器工作台；原因是本项目是 Python-first CLI，Textual 更贴近现代终端产品体验，也更容易随 Python 包分发。
- 假设：用户提到的 “type 什么什么” 指 Textual 类终端 UI 框架；若后续明确需要浏览器前端，可在 `tui web` 另起设计。

## 处理过程

- 新增 `keepa_cli/ui/modern_tui.py`：延迟导入 Textual，定义组件化命令目录、Textual CSS、侧边命令面板、状态卡片、输入栏和结果面板。
- 新增 `keepa-cli tui` / `kc tui` 子命令：显式启动现代 TUI；`--classic` 强制旧标准库界面。
- 默认无命令入口在交互终端优先启动现代 TUI；stdin 非 TTY 时继续走旧管道式 slash 会话，保持脚本兼容。
- `--json tui` 只输出人类界面元数据，不启动交互界面；`--json tui --classic` 标记 `selected_runtime=classic`。
- capabilities 协议增加 `tui`，反映当前入口能力。
- `pyproject.toml` 增加 `textual>=0.86,<2` 依赖；运行时仍有缺依赖 fallback。
- 更新 README 的 TUI 启动、classic fallback 和 JSON 元数据说明。

## 风险与后续

- 风险：全屏视觉已通过 Textual `run_test` 交互 smoke，但未在真实用户终端截图验收；后续如要精修配色和动效，可基于用户截图继续迭代。
- 风险：新增 Textual 依赖会改变 pip 安装体积；已保留缺依赖 fallback，npm wrapper 仍需在目标机器有 Python 依赖环境。
- 风险：Node wrapper 若未指定 `KEEPA_CLI_PYTHON`，可能使用没有 Textual 的系统 Python，此时会自动 fallback 到 classic；release gate 已用当前 `.venv` Python 验证 Node doctor。
- 后续建议：若用户继续要求“浏览器前端框架”，单独设计 `tui web`，不要混入当前终端 TUI 文件。
