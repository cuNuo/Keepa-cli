## 可完善项清单

### P0：治理入口与证据索引

- 项目根缺少 `AGENTS.md`：后续每次进入仓库都需要从 README、memory 和历史 evidence 拼接规则，容易漏掉 `.venv`、双入口、Agent-first、无真实 API 消耗等硬约束。建议新增极简项目 `AGENTS.md`，只放常驻事实、命令入口、memory/evidence/docs 路由和验证清单。
- `evidence/` 缺少 `README.md` 与 `manifest.csv`：当前只有一个历史任务日志和本次任务日志，后续日志增长后不利于快速检索。建议补 evidence 根 README、manifest，并约定逻辑路径与归档策略。
- 缺少 `hooks/README.md` 与项目级 Hook 入口：当前依赖全局 `D:\.codex\hooks\run_relevant_hooks.py`。建议至少补项目级说明，后续若引入 ruff/mypy/文本门禁，再提供项目 `hooks/run_relevant_hooks.py` 包装入口。

### P1：质量门禁与发布可靠性

- CI 仅跑 Windows + Python 3.11 + npm wrapper smoke：建议增加 Linux/macOS 矩阵和 Python 3.12/3.13，提前发现路径、换行和 console script 差异。
- 缺少静态检查配置：当前标准库项目测试覆盖较好，但没有 ruff、mypy/pyright 或 import/order 门禁。建议先引入低噪音 ruff，再逐步增加类型检查。
- `pyproject.toml` 发布元数据偏薄：缺少 authors、classifiers、project.urls、license file、可选 dev 依赖组等。发布 PyPI 前应补齐。
- npm 包未配置 `prepack`/发布前验证脚本：`npm pack --dry-run` 当前可用，但建议把 unittest、schema snapshot、doctor smoke 纳入统一 prepack 或 release check。

### P1：真实 API 与高风险链路

- `graphs.image` live 二进制下载仍未实现：当前明确返回 `live_binary_unsupported`，这是可接受边界，但后续若要完整覆盖 `/graphimage`，需要专用 binary transport、文件输出、MIME/扩展名校验和缓存 TTL 记录。
- 缺少手动触发的 live smoke workflow：现有 CI 正确避免 secret 和真实 token 消耗，但后续需要 `workflow_dispatch` + `KEEPA_API_KEY` + 低成本 cassette 录制/脱敏流程，避免真实链路长期无人验证。
- 高成本命令的预算模型仍偏粗：`offers`、`update=0`、Finder stats、tracking refill 影响等仍适合进一步细化为可解释预算，而不是固定估算。

### P2：产品能力与用户体验

- 本地缓存尚未落地：路线文档已明确 SQLite 缓存和 `--cache-info`，但实现仍未出现。缓存会显著降低误烧 token 风险，也是 Agent 审计能力的重要部分。
- TUI 仍是标准库工作台骨架：当前已复用 command service，但还不是完整搜索、选择、预览、确认、导出工作流。若面向人类长期使用，应继续增强交互效率。
- Agent schema 目前冻结形状而非完整语义：对稳定性有利，但后续可补 schema version、capabilities discovery、错误码枚举和命令能力清单，方便其他 Agent 自动发现。
- 文档缺少贡献与安全入口：建议补 `CONTRIBUTING.md`、`SECURITY.md`、`CHANGELOG.md`，特别是说明真实 Keepa key、cassette 脱敏和发布流程。

### P3：代码结构演进

- `keepa_cli/service.py` 和 `keepa_cli/cli.py` 已较大：当前仍可维护，但继续扩官方 API 时建议按 command family 拆分 service handlers 和 argparse builders，避免单文件继续膨胀。
- `tests/fixtures/` 与 `keepa_cli/fixtures/` 存在重复 fixture：这是打包与测试双场景的现实需要，但后续可以用同步检查或生成脚本避免两边漂移。
- `keepa_cli.egg-info/` 是本地生成目录且已被 `.gitignore` 排除：当前未入库，但每次本地安装会生成，后续任务检查时应继续忽略。

## 超出当前思路的替代路径

- 先做“能力发现协议”：新增 `kc --json capabilities` 输出命令、参数、预算、是否需要确认、是否支持 fixture/live，让 Agent 不再从 README 推断 CLI 能力。
- 先做“release gate”而不是继续加 API：把 unittest、schema snapshot、npm pack、entry smoke、redaction audit 串成一个本地脚本和 CI job，发布质量会比继续堆功能更稳。
- 引入“录制即文档”流程：真实 live smoke 只手动运行，输出脱敏 cassette、自动生成对应 fixture、更新 docs 示例和 schema snapshot，避免真实 API 与离线测试长期脱节。
- 为 token 预算建立策略对象：把预算估算从命令 handler 中抽出，支持 `explain` 字段，未来可按 Keepa 官方规则变更独立更新。
- 设计 cache provenance：缓存不只是命中响应，还记录 endpoint、domain、参数 hash、原始 token cost、生成时间、TTL、数据时间，让 Agent 能解释数据来源。
