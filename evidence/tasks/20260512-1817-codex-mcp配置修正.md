# Codex MCP 配置修正

## 任务目标

修复用户在 Codex 中启用 Keepa MCP 失败的问题，并把 README 中的配置说明改为 Codex 可用的格式。

## 背景与输入

- 用户报错：`MCP client for Keepa failed to start: handshaking with MCP server failed: connection closed: initialize response`。
- 同时出现 `github MCP server is not logged in`，这是独立的 GitHub MCP 登录问题，不影响 Keepa 的 stdio server 命令修正。
- 前一版 README 给了通用 `mcpServers` JSON；Codex CLI / Codex desktop 实际读取 `D:\.codex\config.toml`，或通过 `codex mcp add` 管理。

## 处理过程

1. 运行 `codex mcp list` 与 `codex mcp get Keepa`，确认当前 Keepa 配置存在但 `command` 为空。
2. 直接修复当前 Codex 全局配置：
   - `codex mcp remove Keepa`
   - `codex mcp add Keepa -- D:\github\Keepa-cli\.venv\Scripts\python.exe -m keepa_cli --mcp`
3. 验证 `codex mcp get Keepa` 输出：
   - `command: D:\github\Keepa-cli\.venv\Scripts\python.exe`
   - `args: -m keepa_cli --mcp`
4. 手动向同一命令发送 MCP `initialize` 请求，确认返回 `serverInfo.name=keepa_mcp`。
5. 更新 README / README.zh-CN：
   - 增加 Codex 专用 `codex mcp remove/add/get` 命令。
   - 增加等价 `D:\.codex\config.toml` TOML 片段。
   - 明确 `mcpServers` JSON 只适用于其他 JSON-based MCP client，Codex 不使用该 JSON。

## 验证结果

- `codex mcp get Keepa`：显示 command 与 args 已正确配置。
- `D:\github\Keepa-cli\.venv\Scripts\python.exe -m keepa_cli --mcp` 手动 initialize：通过。
- README / README.zh-CN 中 JSON code block 全部通过 `json.loads`。
- README / README.zh-CN 中 TOML code block 全部通过 `tomllib.loads`。

## 关联产物

- `README.md`
- `README.zh-CN.md`

## 风险与后续

- Codex 当前进程可能已经加载了旧 MCP 配置；修复后需要重启 Codex session 或重启桌面端，让新配置生效。
- GitHub MCP 的 `not logged in` 是另一条远程 MCP 认证问题，按 Codex 提示执行 `codex mcp login github`，或提供有效 `GITHUB_PAT`。

## 结论

Keepa MCP 失败原因是 Codex 中 `Keepa` server 的 `command` 为空。已修复为 `.venv` Python stdio 入口，并把 README 改为 Codex 专用配置优先，避免继续误用通用 JSON。
