## 处理过程

### 1. 分支覆盖缺口审计

- 做了什么：运行 `coverage run --branch -m unittest discover -s tests` 与 `coverage report --show-missing`，逐轮读取 missing branches。
- 为什么这么做：行覆盖 100% 仍可能遗漏条件判断的另一侧；本轮按更严格口径审计。
- 结果：初始 branch coverage 为 99%，存在 56 个 partial branch；补齐后为 6439 statements、2782 branches、0 partial、TOTAL 100%。

### 2. 严格分支测试补齐

- 做了什么：新增 `tests/test_backend_branch_strict_coverage.py`，覆盖 MCP tool call、resource compaction、workflow resolver、KeepaClient fake opener/cache、category/tracking/product command、service fallback、research graph、brief、figures 与 report/cache 边界。
- 为什么这么做：把真实请求相关路径拆成离线可审计的 fake/mock/fixture 场景，避免 live API 和 token 消耗。
- 结果：新增测试单文件通过，完整测试集从 300 增至 308 tests OK。

### 3. 后端边界缺陷修复

- 做了什么：将裸文件输出路径的父目录判断从 `if path.parent` 修为 `if path.parent != Path(".")`；修复 `explain_cache` 读取非 Mapping JSON 时的 `payload.get` 崩溃。
- 为什么这么做：`Path("file").parent` 为 `.`，原判断永远为真，false branch 不可达；非对象 cache payload 应保守返回 unknown，而不是抛出 AttributeError。
- 结果：相关分支变为可达且有测试覆盖；未改变真实 Keepa 请求策略。

### 4. 覆盖率门禁固化

- 做了什么：在 `pyproject.toml` 的 `[tool.coverage.run]` 增加 `branch = true`。
- 为什么这么做：后续普通 `coverage run -m unittest discover -s tests` 默认执行分支覆盖统计，不再依赖人工记得加 `--branch`。
- 结果：配置化 coverage 命令直接通过 100% 分支覆盖门禁。
